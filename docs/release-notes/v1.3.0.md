# javalens v1.3.0 — Single-workspace mode (multi-project)

This release introduces **multi-project workspaces**: a single javalens
process can now hold N Java projects loaded simultaneously, sharing one
Eclipse workspace and one MCP socket. Cross-project search is the default;
agents can scope down to one project per call.

It is the first release on the new path toward replacing the
"one-javalens-process-per-project" model. javalens-manager continues to use
the single-project `load_project` flow for now (no breaking change); the
manager-side wiring to drive multi-project workspaces lands in a later
release. The new tools and the `projectKey` parameter are immediately
usable for anyone speaking MCP directly.

## What's new

### Multi-project workspace API
- New tool `add_project(path)` — append a project to a running workspace
  without clearing existing projects. Returns the derived `projectKey`.
- New tool `remove_project(projectKey)` — drop a project; reassigns the
  default key to the next remaining project deterministically.
- New tool `list_projects()` — returns `defaultProjectKey` plus the array
  of loaded projects with `isDefault` flags.
- `health_check` now emits a `workspace` block with `projectCount` and the
  full per-project list.

### Cross-project search by default
- `get_search_service()` now returns a workspace-scoped `SearchService`
  whose JDT search scope spans every loaded project. `find_references`,
  `search_symbols`, `find_implementations`, and the rest of the indexed
  tools see all projects with no extra parameters.
- The scope is rebuilt automatically on `add_project` / `remove_project` /
  `load_project`.

### Optional `projectKey` parameter on every analysis tool
- All 61 analysis tools accept an optional `projectKey: string` argument
  (visible in `tools/list`). When present, the tool runs against just that
  project's classpath. When absent, tools run cross-project as above.
- Unknown `projectKey` returns an `INVALID_PARAMETER` error pointing the
  caller at `list_projects`.
- The 5 project-management tools (`load_project`, `add_project`,
  `remove_project`, `list_projects`, `health_check`) intentionally do not
  accept `projectKey` — they operate on the workspace as a whole.

### Cross-project file and type lookup (no `projectKey` required)
- `getCompilationUnit(filePath)` and `findType(typeName)` now resolve
  across every loaded project, not just the legacy default. Required for
  agents using file-path-anchored tools (e.g. `get_hover_info`,
  `find_references`) on files that live in non-default projects.

## Behavior preserved
- Single-project usage is unchanged. Tools that pre-date Sprint 10 keep
  reading `service.getJavaProject()` / `service.getSearchService()`; with
  one project loaded the behavior is identical to v1.2.1.
- `load_project` still has "wipe and load one" semantics — use
  `add_project` to grow a workspace incrementally.
- The default project key is the most recently loaded project (set by
  `load_project`) or the first added project if `load_project` has not
  been called.

## For tool authors / direct MCP clients
- `LoadedProject` record exposes `projectKey`, `projectRoot`,
  `javaProject`, `searchService`, `pathUtils`, `loadedAt`,
  `sourceFileCount`, `packageCount`, `packages`, `buildSystem`.
- `IJdtService` adds `defaultProjectKey()`, `getProject(key)`,
  `projectKeys()`, `allProjects()`, `addProject(path)`,
  `removeProject(key)`. All have safe single-project default
  implementations on the interface.
- `ScopedJdtService(delegate, LoadedProject)` is the public adapter that
  tool infrastructure (`AbstractTool`) wraps when `projectKey` is
  present. Workspace-wide queries delegate; single-project getters
  re-point.
- All Sprint 10 helpers stay portable: `java.nio.file.Path`, DOM, and
  `java.util.jar.Manifest` only; no JDT internals leak through helper
  signatures (ADR 0004).

## Compatibility
- Java 21 (built with Temurin 21 in CI).
- Eclipse 4.38 / Tycho 5.0.1.
- Drop-in replacement for v1.2.1 — pull via `javalens-manager` with
  `JAVALENS_RELEASE_REPO=hw1964/javalens-mcp` (or the GitHub Releases
  page).
