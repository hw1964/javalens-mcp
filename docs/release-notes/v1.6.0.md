# javalens v1.6.0 — Ring 1 workspace verification tools

Two new MCP tools that close the agent's *refactor → compile → test → fix* loop. Per-workspace tool count: **60 → 62.** Drop-in upgrade.

## What's new

### Two new MCP tools

| Tool | What it does |
|---|---|
| `compile_workspace([projectKey, minSeverity, includeTaskMarkers])` | Trigger an incremental Java build over every loaded project (or one specific project), refresh local resources first, then aggregate `IMarker` problem markers — compile errors, warnings, plus project-level errors (broken `Require-Bundle`, classpath, manifest). Compilation errors are a *result*, not a tool failure. |
| `run_tests(scope[, framework, timeoutSeconds, vmArgs])` | Launch JUnit / TestNG via JDT-LTK's launching delegate, headless. `scope.kind` is `method` / `class` / `package`. Returns parsed pass / fail / skipped counts plus per-failure stack traces, with bounded stdout/stderr tail capture. Auto-detects JUnit 4 / 5 / TestNG from the project's classpath when `framework="auto"`. |

### Why IMarker, not AST reconcile

`get_diagnostics` (Sprint 9) uses `cu.reconcile(...)` to surface per-file compiler problems via `IProblem`. That's perfect for "what's wrong in this file" but misses cascading errors in untouched files and project-level errors (manifest, classpath, missing `Require-Bundle`). `compile_workspace` runs JDT's `IncrementalProjectBuilder` and reads `IMarker.PROBLEM` from `IResource.findMarkers(...)` — same path the IDE's Problems view uses. One call, full coverage.

### Implementation notes

- `CompileWorkspaceTool` extends `AbstractTool`, walks `service.allProjects()`, calls `project.refreshLocal(IResource.DEPTH_INFINITE, ...)` before each build so the agent's most-recent file edits are always picked up.
- `RunTestsTool` delegates to a new `JUnitLaunchHelper` that builds an `ILaunchConfigurationWorkingCopy` of type `org.eclipse.jdt.junit.launchconfig`, registers a `TestRunListener` for programmatic result collection (`sessionStarted` / `testCaseStarted` / `testCaseFinished`), and captures stdout/stderr through `IProcess.getStreamsProxy()` (capped at 1 MB per stream, last 100 lines surfaced as `stdoutTail` / `stderrTail`).
- The launcher uses **inlined string values** for the launch-config attribute keys (`org.eclipse.jdt.junit.TEST_KIND`, `org.eclipse.jdt.junit.TESTNAME`, `org.eclipse.jdt.junit.CONTAINER`, etc.) instead of importing `JUnitLaunchConfigurationConstants`. The values are forever-stable because they're persisted in user-saved `.launch` configuration XML; inlining sidesteps the awkward "are those constants in the public API surface?" question across Eclipse releases.
- Same target platform (Eclipse 2024-09); three new bundles added to it: `org.eclipse.jdt.junit.core`, `org.eclipse.jdt.junit.runtime`, `org.eclipse.jdt.junit5.runtime`.

## Known limitation — `run_tests` happy-path tests `@Disabled`

The three `run_tests` happy-path tests (`happy_methodScope`, `happy_classScope`, `happy_packageScope`) are `@Disabled` for v1.6.0. Tycho-surefire's headless test runtime doesn't compile our sample-project fixtures (the forked test JVM needs the fixture's classes on disk, and Tycho's test stage doesn't run javac on `test-resources/sample-projects/.../src/test/java`).

Production usage works: when the manager points an agent at a real workspace, `run_tests` launches the same way Eclipse IDE does (the project's actual classpath, the project's compiled classes, the project's own test framework jars). Validation paths — bad scope, missing `packageName`, unknown `kind` — are tested.

Full happy-path coverage lands in **v1.6.1** along with the cross-bundle `compile_workspace` integration test, once the fixture-build pipeline is in place.
