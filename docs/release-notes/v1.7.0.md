# javalens v1.7.0 — Ring 2/3/4 code generation, dependency management, formatter

11 new MCP tools that close the structural-Java gaps the agent previously hand-wrote with `Edit`. Per-workspace tool count: **62 → 73**. Drop-in upgrade.

## What's new

### Ring 2 — code generation (6 tools)

| Tool | What it does |
|---|---|
| `generate_constructor(filePath, line, column, fields[, visibility, callSuper])` | Generate a constructor that initialises the given fields. Visibility selectable; optional `super()` chaining. |
| `generate_getters_setters(filePath, line, column, fields[, kind, visibility])` | JavaBean-style accessors for the given fields. Multi-field; complements `encapsulate_field` (single-field). Boolean fields use `isField()`. Existing accessors are skipped (warning), never overwritten. |
| `generate_equals_hashcode(filePath, line, column, fields)` | `equals(Object)` + `hashCode()` over the given fields. Primitives use `==`, references use `Objects.equals(...)`. `hashCode()` delegates to `Objects.hash(...)`. The `java.util.Objects` import is added if missing. |
| `generate_tostring(filePath, line, column, fields[, style])` | `toString()` in `STRING_CONCATENATION` (default) or `STRING_BUILDER` style. `STRING_FORMAT` and `STRING_BUILDER_CHAINED` arrive in v1.8.x. |
| `override_methods(filePath, line, column[, methods])` | Two modes — query (no `methods` arg → returns `availableMethods[]` listing overridable signatures, excluding `java.lang.Object` by default), and generate (`methods=["describe"]` → `@Override` stubs throwing `UnsupportedOperationException("not yet implemented")`). |
| `generate_test_skeleton(filePath, line, column[, framework, includePrivateMethods])` | Writes a JUnit/TestNG test class to the `src/test/java` mirror of the source package. One `@Test` stub per public method plus `setUp()`. Auto-detects framework via the new shared `FrameworkDetection` helper. |

### Ring 3 — build & dependency management (3 tools, Maven-only)

| Tool | What it does |
|---|---|
| `add_dependency(groupId, artifactId, version[, scope, projectKey])` | Adds a `<dependency>` entry to the project's `pom.xml`. Refuses duplicates (use `update_dependency`). Text-level mutation preserves existing formatting and comments. |
| `update_dependency(groupId, artifactId, newVersion[, projectKey])` | Bumps the `<version>` of an existing dependency in place. Refuses unknown coordinates. |
| `find_unused_dependencies([projectKey])` | Read-only: lists declared dependencies whose packages don't appear in any source-import. Heuristic match (`groupId` prefix or `artifactId` as dotted suffix); false positives possible — treat as suggestions. |

Gradle support is **explicitly deferred to v1.8.x**: tools detect non-Maven projects and return `INVALID_PARAMETER` with a clear message rather than guessing.

### Ring 4 — formatter / workflow polish (2 tools)

| Tool | What it does |
|---|---|
| `format(scope[, dryRun])` | Apply the JDT formatter at file / package / project / workspace scope. Honours the project's own `.settings/org.eclipse.jdt.core.prefs`. `dryRun: true` returns a diff sample without writing files. |
| `optimize_imports_workspace([scope])` | Workspace fan-out of import optimisation: removes unused imports and sorts (`java.*` → `javax.*` → others → static) across every loaded project. Mutates files in place; idempotent. |

## Implementation notes

- **No `org.eclipse.jdt.ui` dependency.** The Ring 2 codegen tools build source via `ASTRewrite` and the JDT-DOM `AST` factory directly — same Sprint 11 LTK pattern. The "obvious" path through `org.eclipse.jdt.internal.corext.codemanipulation.GenerateConstructorOperation` / `GenerateHashCodeEqualsOperation` / etc. requires `org.eclipse.jdt.ui`, which is the chunky GUI bundle and not on our target platform.
- **`FrameworkDetection`** at `org.javalens.mcp.tools.shared.FrameworkDetection` (new) is the Sprint 12 `RunTestsTool.resolveFramework` extracted to a public static helper. `RunTestsTool` now delegates; `generate_test_skeleton` reuses the same logic. No behavioural change to `run_tests`.
- **Maven dep tools** use `javax.xml.parsers.DocumentBuilder` for read-only inspection and text-level regex/string manipulation for in-place edits. M2E is intentionally **not** required — keeps tools usable in any Tycho runtime, not just one with M2E auto-import. Richer M2E-resolved-classpath inspection lands as a v1.8.x enhancement.
- **`OptimizeImportsWorkspaceTool`** parses each CU's AST, walks **only the type declarations** (not the import statements themselves), and uses binding-resolved type references to detect unused imports. Visiting the full AST mistakenly counts the import declaration's own qualified name as a use of the type — guarding against that is the subtle correctness bug the workspace fan-out adds beyond per-file `organize_imports`.

## Known limitations

- `generate_test_skeleton` auto-detect happy-path test is `@Disabled` — Tycho-surefire's test runtime doesn't resolve external Maven deps onto JDT's classpath, so `FrameworkDetection.detect("auto", project)` finds nothing. Production usage (real workspace with M2E-resolved classpath) detects correctly. Same fixture-build gap that has Sprint 12's `run_tests` happy-paths `@Disabled`. Tracked in `docs/upgrade-checklist.md`.
- `find_unused_dependencies` is heuristic: declared deps whose `groupId` is a prefix of any source import (or whose `artifactId` appears dotted-suffix in any import) are deemed used. False positives are documented; treat results as suggestions, not as gospel.
