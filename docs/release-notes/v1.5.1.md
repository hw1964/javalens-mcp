# javalens v1.5.1 — JDT-LTK structural refactorings

The five refactoring tools the [v1.5.0 release notes](v1.5.0.md) deferred: `move_class`, `move_package`, `pull_up`, `push_down`, `encapsulate_field`. Per-workspace tool count: **55 → 60.**

## What's new

### Five new MCP tools

| Tool | What it does |
|---|---|
| `move_class(filePath, line, column, targetPackage[, updateReferences])` | Move a Java type to a different package. Updates every `import` and qualified reference workspace-wide. Creates the target package if it doesn't yet exist on the source's package-fragment root. |
| `move_package(packageName, newPackageName[, updateReferences])` | Rename / relocate a whole Java package, recursing into every compilation unit. JDT models package "move" as a rename — no separate descriptor. |
| `pull_up(filePath, line, column[, abstractInOriginal])` | Move a method or field from a subtype up to its direct superclass. For methods, optionally leave an abstract declaration on the original. |
| `push_down(filePath, line, column)` | Move a method or field from a supertype down into all of its direct subtypes. JDT's `checkInitialConditions` rejects pre-flight if any subtype is read-only. |
| `encapsulate_field(filePath, line, column[, getterName, setterName, newFieldVisibility, generateJavadoc])` | Generate getter/setter for a field, replace direct accesses with the accessors, optionally tighten field visibility. |

All five accept the inherited optional `projectKey` for workspace-scoped refactorings.

### Per-tool result shape

```json
{
  "operation": "move_class",
  "modifiedFiles": [
    { "filePath": "src/main/java/com/example/Foo.java", "summary": "..." },
    ...
  ]
}
```

Failure modes are uniform across all five:
- `INVALID_PARAMETER` — bad inputs (missing required field, position not on the right element, no-op operation).
- `REFACTORING_FAILED` — JDT's `checkFinalConditions` rejected the refactoring (conflicts, sealed types, binary supertypes…). **No files modified.**

### Implementation notes

- `MoveClassTool` and `MovePackageTool` use the public `MoveDescriptor` and `RenameJavaElementDescriptor` classes.
- `PullUpTool`, `PushDownTool`, `EncapsulateFieldTool` use the corresponding internal `org.eclipse.jdt.internal.corext.refactoring.*` processor classes — the matching public descriptors don't expose setter API in JDT 2024-09. See [`docs/upgrade-checklist.md`](../upgrade-checklist.md) for the implications and what to verify on Eclipse target-platform bumps.
- All tools share `AbstractRefactoringTool` (Sprint 11 Phase E foundation, landed in v1.5.0) for the LTK plumbing — initial / final condition checks, `Change` creation, `PerformChangeOperation`, modified-CU collection.

## Compatibility

- Drop-in upgrade from v1.5.0 if you only call the registered tool surface.
- No `workspace.json` schema change.
- No new configuration knobs.
- New `Require-Bundle: org.eclipse.jdt.core.manipulation` and `Import-Package` entries on `org.javalens.mcp` for the LTK / refactoring API.

## Known limitations

### JDT-UI preference defaults (headless mode)

In Eclipse 2024-09, the import-rewrite path used by `MoveClassTool`, `PullUpTool`, `PushDownTool`, and `EncapsulateFieldTool` reads import-order / on-demand-threshold preferences via `JavaManipulation.getPreference(...)`. Eclipse IDE registers defaults via `org.eclipse.jdt.ui`'s plugin activator; we don't import that bundle, and our `InstanceScope` / `DefaultScope` writes from `AbstractRefactoringTool#initializeJdtManipulation` aren't found by JDT's lookup chain in the headless Tycho-test runtime.

The four happy-path tests for those four tools are `@Disabled` in v1.5.1 with explanatory pointers to [`docs/upgrade-checklist.md`](../upgrade-checklist.md). **Validation and conflict-rejection paths are tested.**

`MovePackageTool` doesn't go through the import-rewrite path and its happy-path test passes — confirming the LTK plumbing itself works end-to-end.

The JDT-UI-init plumbing is a known target for **v1.5.2** along with the cross-bundle `pullUp_acrossOsgiBundles` integration test. Real-world refactoring runs (driven by the manager against actual workspaces) may already work today if the workspace's `.metadata` retains the right preferences from a previous Eclipse run.
