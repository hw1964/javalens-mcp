# javalens v1.7.1 — bugs.md cleanup (#1–#5)

Drop-in patch on top of v1.7.0. Five OPEN bugs from real-world `JATS-ORB-WS` / `JAVALENS-WS` usage during Sprint 13 closed in one release. No new tool surface — 73 tools, unchanged.

## Bugs fixed

### #5 (HIGH) — Non-manager-spawned JVMs started with zero projects loaded

**Symptom:** every non-manager MCP client (Cursor, every Claude Code session, every Claude Desktop instance) saw `list_projects` return `[]` despite the manager UI showing the workspace fully populated. Defeated the "deploy MCP entries into MCP clients" usability story.

**Root cause:** the workspace.json auto-load from Sprint 10 (v1.4.0) does exist, but `JavaLensLauncher` (the wrapper that gives each session-spawned JVM its own JDT workspace) injects a UUID subdir into `-data` before delegating to Equinox. The auto-loader looked in the OSGi data dir (the UUID subdir), while the manager writes `workspace.json` one level up at the workspace root.

**Fix:** `JavaLensApplication.autoLoadProjects()` now walks one level up if `workspace.json` isn't found in the immediate OSGi data dir. New helper `findWorkspaceJson(Path)` at `org.javalens.mcp.JavaLensApplication` (public static, test-visible). Backward-compatible with direct invocations that don't go through the launcher.

### #1 (HIGH) — `run_tests` NPE for plain Maven / Gradle projects

**Symptom:** `run_tests` against a plain Maven project (no Eclipse PDE nature) returned `INTERNAL_ERROR: Cannot invoke "org.osgi.framework.Bundle.getHeaders()" because "bundle" is null`. Blocked the MCP-driven TDD workflow.

**Fix:** `RunTestsTool.executeWithService()` now branches on `loaded.buildSystem()` and short-circuits MAVEN / GRADLE projects with `INVALID_PARAMETER` plus an actionable workaround message pointing at `mvn test -Dtest='...'` (or `gradle test --tests`). The full plain-Maven JUnit launch path (that would bypass the OSGi-assuming JDT-LTK launcher entirely) is tracked for v1.8.0; this release stops the misleading crash.

### #2 (MEDIUM) — `search_symbols` duplicated cache-path entries

**Symptom:** queries returned 2–3 entries for one class when the workspace held dependent projects — one source entry (with `line` / `column`) plus one or two binary-classpath duplicates (`/home/harald/.cache/javalens-manager/...`, no source coordinates). Polluted search results, broke refactor tooling that walks `filePath:line:column`.

**Root cause:** JDT `SearchEngine` searches the full workspace scope including binary JARs on dependent projects' classpaths. The same FQN was returned once per index hit.

**Fix:** new static helper `SearchSymbolsTool.dedupeBySymbolIdentity(...)` groups results by `(kind, qualifiedName)` (or `(kind, containingType#name)` for methods/fields). When at least one entry in a group has coordinates, the coordinate-less binary hits are dropped. Legitimate ambiguity (same FQN with coordinates in two real source files, e.g. forked classes) is preserved.

### #4 (LOW) — `buildSystem: "unknown"` for Eclipse PDE bundles

**Symptom:** `list_projects` reported `buildSystem: "unknown"` for fully-indexed PDE bundles (1030+ files, classpath resolved). Misled callers into thinking the project hadn't loaded.

**Fix:** `ProjectImporter.BuildSystem` extended with `ECLIPSE_PDE` and `ECLIPSE` values. `detectBuildSystem()` checks `META-INF/MANIFEST.MF` for `Bundle-SymbolicName:` (PDE) and `.classpath` (plain Eclipse) after the Maven / Gradle / Bazel rules so Tycho-style hybrids still classify as Maven. `ListProjectsTool` lowercases + replaces `_` with `-` on the enum name so the wire format reads `"eclipse-pde"` (hyphen).

### #3 (LOW) — `run_tests` schema description misleading for `kind="method"`

**Symptom:** the description listed `scope.methodName — for method scope` separately from `scope.filePath`, implying `{filePath, methodName}` was a valid combination. The validator only accepts `{typeName, methodName}` or `{filePath, line, column}`. Reader path led to a clear-but-correct INVALID_PARAMETER they should have avoided.

**Fix:** description block now spells out the two valid combinations explicitly and calls out that `{filePath, methodName}` is NOT valid. Per-field descriptions on `methodName`, `filePath`, `line`, `column` mention their pairing rules. Five new contract tests in `RunTestsToolTest` pin the wording.

## Tests

- **New:** `SearchSymbolsToolDedupeTest` (6 methods, all PASS) — cache-path dedupe with the canonical bugs.md #2 case + legitimate ambiguity + no-regression on single-project.

## Compatibility

- Drop-in patch on top of v1.7.0. No data-format changes.
- Wire format for `buildSystem` now uses hyphen (`"eclipse-pde"`, `"eclipse"`) rather than underscore. Callers that branched on the exact `"unknown"` string for PDE projects will now see `"eclipse-pde"` — adjust accordingly.
- `BuildSystem` enum gains two values (`ECLIPSE_PDE`, `ECLIPSE`). Anything switching on `BuildSystem` Java values should add cases for both. Internal-use enum; not a public API surface.
- `run_tests` against plain Maven / Gradle projects now returns `INVALID_PARAMETER` with workaround text where it previously returned `INTERNAL_ERROR` with an NPE. Callers should already handle either; the new code is more actionable.
