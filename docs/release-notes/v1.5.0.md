# javalens v1.5.0 — Detection-matrix completion + tool-surface consolidation

Sprint 11 closes correctness gaps for three of the four supported project layouts (Tycho-packaged Maven, PDE bundles with `Require-Bundle` deps, Gradle), and shrinks the per-service MCP tool count so multi-workspace agent setups have headroom under their 100-tool budgets.

## What's new

### Tycho-aware Maven detection (Phase A)

Projects whose `pom.xml` declares a Tycho packaging type — `eclipse-plugin`, `eclipse-test-plugin`, `eclipse-feature`, `eclipse-repository`, `eclipse-update-site`, `eclipse-target-definition` — no longer run `mvn dependency:build-classpath`. That goal returns wrong/empty results for Tycho projects because their dependencies come from the target platform plus `MANIFEST.MF`, not pom `<dependencies>`.

`ProjectImporter` now reads `<packaging>` and gates the Maven shell-out accordingly. Tycho projects rely on `.classpath kind="lib"` entries plus `Require-Bundle` resolution against the workspace bundle pool (Phase B).

New public helpers in `org.javalens.core.project.ProjectImporter`:
- `readPomPackaging(Path pomXml) -> Optional<String>`
- `isTychoPackaging(String packaging) -> boolean`
- `isTychoProject(Path projectPath) -> boolean`

### Workspace bundle pool — `Require-Bundle` resolution (Phase B)

PDE bundles (Tycho-packaged or pure Eclipse PDE) loaded into the same workspace now have their inter-bundle `Require-Bundle` declarations resolved to project-typed classpath entries. Cross-bundle `find_references`, `find_implementations`, and other JDT-driven analyses work transparently across bundle boundaries.

- New helpers in `ProjectImporter`:
  - `readManifestSymbolicName(Path projectRoot) -> Optional<String>` — reads `Bundle-SymbolicName`, strips `;singleton:=true` and other directives.
  - `readManifestRequireBundle(Path projectRoot) -> List<String>` — parses the comma-separated `Require-Bundle` header (with multi-line continuation support via `java.util.jar.Manifest`).
- `WorkspaceManager` carries a thread-safe `Bundle-SymbolicName → IProject` map, populated by `JdtServiceImpl` as each project loads. `resolveBundle(symbolicName) -> Optional<IJavaProject>` looks bundles up at classpath-configuration time.
- `addDependencyEntries` iterates `Require-Bundle` and appends `JavaCore.newProjectEntry` for each sibling resolvable through the pool. Externally-resolved bundles (system bundles in the target platform, IDE-managed deps) stay unresolved with a debug log.

**Load ordering note:** a project's `Require-Bundle` resolves only against bundles already in the pool when its classpath is configured. Load PDE bundles bottom-up (depended-on first); the manager's `workspace.json` sequencing honours this naturally.

### Gradle Tooling API integration (Phase C)

`getGradleDependencies` is no longer empty — Gradle projects now have their actual source roots and resolved jar dependencies extracted via the [Gradle Tooling API](https://docs.gradle.org/current/userguide/tooling_api.html), not by walking `build/classes/java/...` after the fact.

- New public helper `ProjectImporter.readGradleProjectModel(Path projectPath) -> Optional<GradleProjectModel>` using `GradleConnector` + the `EclipseProject` model. Returns `Optional.empty()` on any failure (no Gradle distribution available, daemon launch failure, missing java plugin) so callers can fall back gracefully — heuristic walking of `build/classes/java/{main,test}` remains as a backstop.
- `getAllSourcePaths` picks up Gradle-declared `sourceSets.main.java.srcDirs` overrides on top of the existing heuristic.
- `org.gradle:gradle-tooling-api:8.10` ships embedded under `org.javalens.core/lib/` and is referenced from MANIFEST.MF's `Bundle-ClassPath` header. The bundle's classloader resolves the API at runtime; `build.properties`'s `jars.extra.classpath` + `bin.includes` wires the same jar onto the compile classpath and into the built bundle. No target-platform reconfig, no bnd-tools wrap.

**Runtime requirement:** the Tooling API needs a Gradle distribution at runtime. On first call it downloads one (~150 MB into `~/.gradle/caches/dists`); subsequent calls use the cache. Test environments without network access can opt out with `-Djavalens.skip.gradle=true`.

### Tool-surface consolidation: 66 → 55 (Phase D)

13 narrow `find_*` tools collapsed into 2 parametric tools so multi-workspace agent setups retain headroom under Antigravity's ≈100-tool registration cap. With GitKraken's MCP shim already dropped, two active workspaces now total about 110 tools — comfortable.

#### `find_pattern_usages(kind, query[, maxResults])`

Replaces five narrow type-anchored search tools:

| `kind`           | replaces                       |
|------------------|--------------------------------|
| `annotation`     | `find_annotation_usages`       |
| `instantiation`  | `find_type_instantiations`     |
| `type_argument`  | `find_type_arguments`          |
| `cast`           | `find_casts`                   |
| `instanceof`     | `find_instanceof_checks`       |

#### `find_quality_issue(kind[, filePath, query, maxMethods, maxFields, maxLines, maxResults])`

Replaces eight narrow code-quality tools:

| `kind`            | replaces                       |
|-------------------|--------------------------------|
| `naming`          | `find_naming_violations`       |
| `bugs`            | `find_possible_bugs`           |
| `unused`          | `find_unused_code`             |
| `large_classes`   | `find_large_classes`           |
| `circular_deps`   | `find_circular_dependencies`   |
| `reflection`      | `find_reflection_usage`        |
| `throws`          | `find_throws_declarations`     |
| `catches`         | `find_catch_blocks`            |

For `kind=throws` and `kind=catches`, the unified `query` parameter aliases to the underlying tools' `exceptionType` parameter — the consolidated tool's input shape stays uniform without rewriting the analysers.

#### Schema discoverability

Both parametric tools declare typed `kind` enums in their input schema with per-kind descriptions, so agents can discover what's available through `tools/list` without trial and error.

#### Breaking change for direct MCP clients

The 13 narrow tool *registrations* are gone. The narrow tool *classes* remain in the package as the analysis implementations the parametric tools delegate to. Any MCP client that called e.g. `find_annotation_usages` directly must move to `find_pattern_usages(kind="annotation", ...)`. javalens-manager already calls only the registered surface — no manager-side changes required for v1.5.0 / manager v0.11.0.

`find_method_references` stays as-is (different parameter shape — anchor-based, not type-anchored).

### Phase E foundation (preparation for v1.5.1)

`org.eclipse.jdt.core.manipulation` and `org.eclipse.ltk.core.refactoring` now ship in the target platform; `org.javalens.mcp` imports the relevant packages. A new `AbstractRefactoringTool` encapsulates the LTK plumbing — initial / final condition checks, `Change` creation, `PerformChangeOperation`, modified-CU collection — so the five concrete refactoring tools can stay small (~80–120 LOC of descriptor wiring each).

The five concrete tools (`move_class`, `move_package`, `pull_up`, `push_down`, `encapsulate_field`) are deferred to **v1.5.1** per the Sprint 11 plan's cut-line. Landing them on the foundation that's already in v1.5.0 means v1.5.1 won't need another target-platform bump.

## Tool count after v1.5.0

- Per-service tools: **66 → 55**, leaves ~45 slots free under the 100-tool cap.
- After v1.5.1 adds the 5 refactoring tools: **60**, ~40 slots free.

## Compatibility

- Drop-in upgrade from v1.4.0 if you only use the manager and registered tools.
- **Breaking** for direct MCP clients calling the 13 narrow `find_*` tool names.
- No `workspace.json` schema change.
- Existing v1.4.0 / v1.3.0 fixtures and tests stay green.

## Tests
