# javalens v1.4.0 — Live workspace.json file-watch

Builds on **v1.3.0**'s multi-project workspace and exposes it to a manager
that can grow or shrink the workspace **without restarting the javalens
process**. v1.4.0 watches a `workspace.json` file in the Eclipse `-data`
directory, picks up additions and removals through `java.nio.file.WatchService`,
and reflects the change in the live MCP service. Agents see the new project
on their very next tool call. No reload, no mcp.json rewrite, no session
restart.

## What's new

### `WorkspaceFileWatcher`
- New class `org.javalens.core.workspace.WorkspaceFileWatcher`.
- Reads `<data-dir>/workspace.json` and loads listed projects on startup
  (`loadProject` for the first, `addProject` for the rest).
- Watches the data dir for ENTRY_CREATE / ENTRY_MODIFY events on
  `workspace.json` and reconciles the live workspace state when the file
  changes — adds new paths, removes paths that disappeared, leaves
  unchanged ones alone.
- Daemon thread; closes cleanly when the application stops.

### `workspace.json` schema
```json
{
  "version": 1,
  "name": "jats",
  "projects": [
    "/abs/path/to/project1",
    "/abs/path/to/project2"
  ]
}
```
The manager writes this file atomically (write-temp + rename) so the watcher
never observes a half-written file.

### Startup precedence in `JavaLensApplication`
1. If `<data-dir>/workspace.json` exists → load via `WorkspaceFileWatcher`
   (the manager-driven path, recommended).
2. Otherwise → fall back to the legacy `JAVA_PROJECT_PATH` env var
   (single-project, manual launches).

Behavior is identical to v1.3.0 for users who don't write a `workspace.json`.

## Behavior preserved
- All v1.3.0 MCP tools work unchanged: `add_project`, `remove_project`,
  `list_projects`, `health_check` workspace block, optional `projectKey`
  scope-down on every analysis tool, workspace-scoped cross-project
  search.
- Agent-driven mutations via `add_project` / `remove_project` MCP calls
  continue to work; the file-watch and the agent-driven paths are
  complementary — both feed the same `JdtServiceImpl`.

## For tool authors / direct MCP clients
- `WorkspaceFileWatcher(jdtDataDir, JdtServiceImpl)` is the public
  constructor. Call `start()` to perform the initial load and arm the
  watcher; call `close()` (or use try-with-resources) to stop.
- The watcher operates entirely on `IJdtService.{addProject, removeProject,
  allProjects, projectKeys}` — same surface that agents and the manager
  use over MCP.
- `org.javalens.core` now imports `com.fasterxml.jackson.{core, databind}`.

## Tests
- 385 tests pass across the suite (381 + 4 new).
- `WorkspaceFileWatcherTest`: initial-load, add-on-file-change,
  remove-on-file-change, malformed-json-survives.

## Compatibility
- Java 21 (built with Temurin 21 in CI).
- Eclipse 4.38 / Tycho 5.0.1.
- Drop-in replacement for v1.3.0. Existing `JAVA_PROJECT_PATH` users see
  no change. javalens-manager v0.10.4+ is the first manager release that
  drives this — see [`hw1964/javalens-manager`](https://github.com/hw1964/javalens-manager).
